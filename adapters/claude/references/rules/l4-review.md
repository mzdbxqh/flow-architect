# L4 审查规则

L4 子流程架构审查规则。

---

## FA-L4-001: 4D 边界合规性检查

**严重级别**: BLOCKER
**确定性**: 否

### 说明

每个 L4 之间的拆分点必须对应一个 4D 事件。4D 拆分标准如下:

| 代码 | 标准 | 定义 |
|------|------|------|
| D1 | 决策点 | 需要管理者/专家做正式判断或批准 |
| D2 | 交付物交接 | 交付物在不同角色或系统之间正式移交，责任主体转移。同一角色内的自检、复核、审核不产生 D2 |
| D3 | 领域切换 | 同一组织内从一个专业领域切换到另一个 |
| D4 | 时限门控 | 存在明确的时间截止要求 |

如果一个步骤同时满足多个 4D 标准，应全部标注（如"D1+D2"）。当 L4 边界无明确 4D 事件但存在职责重大变化、风险等级跃升等业务理由时，标记为"业务判断"而非"不通过"，由用户最终裁决。

### 检查步骤

1. 对每个 L4 边界，检查是否存在对应的 4D 事件。
2. 验证标记的 D1/D2/D3/D4 类型是否成立。
3. 如果边界无 4D 依据，判定为"不通过"或"业务判断"（需提供业务理由）。
4. 输出每个边界的合规性判定。

### 证据要求

- L4 边界标识（如 L4-01 -> L4-02）
- 对应的 4D 事件描述
- 4D 类型（D1/D2/D3/D4）
- 判定结果（通过/不通过/业务判断）

## FA-L4-002: 4D 事件归属检查

**严重级别**: CRITICAL
**确定性**: 否

### 说明

4D 事件本身属于前一个 L4 的最后一步，事件之后开启新的 L4。这是边界归属的核心规则。例如，如果"部长审批"是 L4-01 和 L4-02 之间的 4D 事件（D1），则该审批步骤应归属于 L4-01 的末尾。

### 检查步骤

1. 对每个 L4 边界上的 4D 事件，确认其归属于前一个 L4 的最后一步。
2. 如果 4D 事件被错误地归属到新 L4 的第一步，标记归属错误。
3. 验证边界位置标注是否符合"前 L4 末尾 -> 新 L4 开始"的格式。

### 证据要求

- 4D 事件所在步骤的步骤编号
- 当前归属的 L4 编号
- 正确归属应为的 L4 编号

## FA-L4-003: 末端组织边界检查

**严重级别**: CRITICAL
**确定性**: 否

### 说明

核心约束: 一个 L4 内可有不同角色（执行者/审核者/批准者），但必须属于同一末端组织。跨角色不跨末端组织是 L4 拆分的基本原则。

末端组织包括四种理解角度:
- 职能型末端组织: 科/系级组织单元（如生产管理科、质量检验科）
- 临时型组织: 项目组（如 XX 项目组）
- 流程型组织: 跨职能团队（如敏捷团队、产品团队）
- 流程 Owner 管辖范围: 流程负责人管辖边界

### 检查步骤

1. 列出每个 L4 内所有步骤涉及的组织。
2. 验证所有步骤是否属于同一末端组织。
3. 如果存在跨组织步骤，检查是否属于五类允许的跨组织例外（见 FA-L4-004）。
4. 非例外的跨组织 L4 必须标记为不通过，并建议拆分。
5. 跨组织 L4 必须标注 Owner 组织。

### 证据要求

- L4 编号
- L4 内各步骤所属的末端组织
- 是否存在跨组织情况
- 若跨组织，属于哪类例外（或标记违规）

## FA-L4-004: 跨组织例外审查

**严重级别**: MAJOR
**确定性**: 否

### 说明

原则上 L4 不跨末端组织，但以下五类例外允许跨组织（须标注 Owner 组织）:

| # | 例外类型 | 说明 |
|---|---------|------|
| 1 | 专家评审 | 合同会签、技术评审 |
| 2 | 资源调度 | 跨产线借调、跨部门支援 |
| 3 | L3 衔接点 | 跨组织移交（L3 之间的衔接） |
| 4 | 知识传递 | 新工艺导入、培训指导 |
| 5 | 外部接口 | 供应商准入评审、客户验收 |

跨组织例外优先于 4D 拆分规则。当一个 4D 事件所在位置属于五类跨组织例外之一时，该 4D 事件不作为拆分依据，但仍需在 4D 事件清单中标注，并在"边界位置"列注明"被[例外类型]覆盖"。

### 检查步骤

1. 识别所有跨组织的 L4。
2. 对每个跨组织 L4，判断是否属于五类允许的例外之一。
3. 如果不属于任何例外，标记为违规。
4. 对属于例外的跨组织 L4，验证是否已标注 Owner 组织。
5. 检查被例外覆盖的 4D 事件是否在事件清单中有所标注。

### 证据要求

- 跨组织 L4 编号
- 涉及的组织列表
- 适用的例外类型（或标记违规）
- Owner 组织标注情况

## FA-L4-005: 数量校验

**严重级别**: MAJOR
**确定性**: 是

### 说明

L4 拆分数量应符合参考范围:

| 场景 | 参考范围 | 超出处理 |
|------|---------|---------|
| 简单 L3 | 2-4 个 L4 | 超出需审视是否有过度拆分 |
| 中等 L3 | 4-8 个 L4 | 超出需审视是否有过度拆分 |
| 复杂 L3 | 8-15 个 L4 | 超过"参与职能数 x3"需审视 |

超过 15 个 L4 必须审视，检查是否有过度拆分（将同一职责的连续步骤误拆为多个 L4）。低于参考范围下限也需审视，检查是否有过度合并（将不同职责的步骤强行合并在一个 L4）。数量异常时，逐个检查 L4 边界的 4D 依据是否充分。

### 检查步骤

1. 统计 L4 总数量。
2. 判断 L3 复杂度（简单/中等/复杂）并确定参考范围。
3. 如果数量超出参考范围，标记为 WARNING 并建议审视。
4. 如果数量超过 15，标记为 BLOCKER 并要求审视。
5. 逐个检查数量异常情况下的 L4 边界 4D 依据。

### 证据要求

- L4 总数量
- L3 复杂度判定
- 参考范围
- 超出时的审视建议

## FA-L4-006: 等待步骤验证

**严重级别**: MAJOR
**确定性**: 否

### 说明

纯等待步骤占比过高属于非增值节点。合理性检查要求统计纯等待步骤数占总步骤数的比例，超过 20% 标记 WARNING。

### 检查步骤

1. 识别所有具有等待/挂起/待处理语义的步骤（通过名称或属性判断）。
2. 计算等待步骤数占总步骤数的比例。
3. 如果比例超过 20%，标记为 WARNING。
4. 检查等待步骤是否具备恢复条件、超时设置和升级路径。

### 证据要求

- 等待步骤标识和名称
- 总步骤数
- 等待步骤占比
- 恢复条件、超时、升级路径的具备情况

## FA-L4-007: 重复录入检测

**严重级别**: MAJOR
**确定性**: 是

### 说明

同一数据字段在不同步骤中被重复录入属于非增值节点。重复录入增加了出错风险和人工成本，应尽可能避免。

### 检查步骤

1. 扫描所有步骤的输入/输出字段。
2. 检测是否存在同一数据字段在不同步骤中被录入的情况。
3. 出现即标记 WARNING。

### 证据要求

- 重复录入的字段名称
- 涉及的步骤标识
- 重复录入的具体场景

---

## FA-L4-008: 系统切换检查

**严重级别**: CRITICAL
**确定性**: 否

### 说明

一次 L4 内切换系统次数过多属于非增值节点。超过 3 次系统切换标记 WARNING。单个 L4 步骤内涉及多个系统时应有明确的系统间交接文档。从一个 IT 系统切换到另一个的中途执行应考虑拆分。

### 检查步骤

1. 统计每个 L4 内的系统切换次数。
2. 如果单个 L4 内切换系统超过 3 次，标记为 WARNING。
3. 对每个涉及多系统的步骤，验证是否有明确的系统间交接文档。
4. 对步骤内中途切换系统的情况，标记为需拆分。

### 证据要求

- L4 编号
- 涉及的系统列表
- 系统切换次数
- 系统间交接文档的具备情况

## FA-L4-009: 不必要审批检测

**严重级别**: MINOR
**确定性**: 否

### 说明

对每个 D1 标记（审批步骤），需要审视其必要性: "不批会怎样？"。需区分两类审批:
- **法规强制审批**: 安全检查、合规审查等，标注"保留"，不可砍
- **管理惯性审批**: 所有申请都批、从无驳回，可审视优化

无实质风险的管理惯性审批标记为 REVIEW。

### 检查步骤

1. 识别 L4 流程中所有 D1 标记的审批步骤。
2. 对每个审批步骤，评估其是否为法规强制审批或管理惯性审批。
3. 检查同一审批者是否在之前已对相同内容审批过。
4. 检查审批步骤是否有拒绝/驳回路径。
5. 对管理惯性审批标记为 REVIEW。

### 证据要求

- 审批步骤标识和分配角色
- 审批类型（法规强制/管理惯性）
- 同角色的前置审批步骤
- 是否存在拒绝/驳回路径

## FA-L4-010: L4 步骤完整性

**严重级别**: CRITICAL
**确定性**: 是

### 说明

每个 L4 步骤必须具备以下完整信息:

| 要素 | 说明 |
|------|------|
| 步骤编号 | 按顺序编号 S1, S2, S3... |
| 步骤描述 | 做什么 |
| 执行角色 | 谁做（不可使用人名，应为角色名称） |
| 所属组织 | 属于哪个末端组织 |
| 输入/输出 | 交付物是什么 |
| 时间约束 | 是否有截止要求 |

### 检查步骤

1. 对每个 L4 步骤，验证是否存在: 步骤编号、步骤描述、执行角色、所属组织、输入、输出。
2. 验证执行角色引用的是有效角色而非具体人名。
3. 验证输入和输出名称为非空字符串。
4. 标记缺少一个或多个必填字段的步骤。

### 证据要求

- 步骤标识
- 各必填字段的存在情况（存在/缺失）
- 缺失字段的具体名称
